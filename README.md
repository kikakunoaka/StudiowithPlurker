# 周邊工作室評價站

純前端網站（HTML/CSS/JS），資料即時從你的 Google 試算表讀取，不需要後端、不需要 API Key，可直接部署在 GitHub Pages。

## 最新調整（第九輪）

- **經驗分享新增「工作室回覆」**：詳情頁每則經驗分享卡片下方，如果試算表「經驗分享」分頁 I 欄（表頭文字「工作室回覆」）有填內容，會多顯示一個「工作室回覆」區塊，用不同底色（橘色系）跟原本的分享內容做區隔，方便訪客一眼看出這是工作室的官方回應。
  - 欄位對應寫在 `config.js` 的 `REVIEW_FIELDS.STUDIO_REPLY`，跟其他欄位一樣是照表頭文字抓資料，該欄留空就不會顯示回覆區塊。
  - 回覆內容一樣會跑過 `formatTextWithLinks()`，裡面如果有網址會自動變成可點擊連結。
  - 樣式在 `style.css` 的 `.review-reply` / `.review-reply-label` / `.review-reply-content`。
  - 這個回覆欄位目前是**唯讀**：網站本身不能讓工作室直接在頁面上輸入回覆，你需要自己到試算表的 I 欄手動貼上工作室的回覆文字，存檔後網站會自動抓到最新內容。

## 最新調整（第八輪）

- **返圖網址自動判斷是否為圖片**：詳情頁「體驗回報」的返圖網址（`返圖網址(1)~(3)`），現在會先判斷網址副檔名是不是常見圖片格式（`.jpg` `.jpeg` `.png` `.gif` `.webp` `.bmp` `.svg`）：
  - 是圖片 → 跟原本一樣顯示縮圖，點擊在新分頁開啟原始網址。
  - 不是圖片（例如雲端硬碟分享連結、短網址等抓不到副檔名的網址）→ 改顯示一個「返圖連結」按鈕，點擊在新分頁開啟該網址，避免縮圖區塊出現破圖。
  - 判斷邏輯統一寫在 `assets/js/gsheet.js` 的 `isImageUrl()`。因為是純前端、沒有後端可以真的檢查檔案的 Content-Type，只能靠網址副檔名判斷，如果分享連結本身沒有副檔名（例如純看 ID 的雲端硬碟連結），就會被歸類成「不是圖片」而顯示連結按鈕。
- **首頁工作室卡片新增排序功能**：篩選區塊新增「排序方式」下拉選單，可選：
  - **評價多→寡**（預設）：依該工作室體驗回報的**則數**由多到少排序。
  - **評價高→低**：依該工作室體驗回報的**平均分數**由高到低排序。
  - 完全沒有體驗回報的工作室，則數與平均分數都視為 0，排序時會排在最後面。邏輯在 `assets/js/app.js` 的 `applyFilters()` 裡。
- **詳情頁體驗回報新增排序功能**：「體驗回報」標題下方新增排序下拉選單，可選：
  - **評價新→舊**（預設）：依「時間戳記」欄位由新到舊排序（跟原本行為一樣）。
  - **評價高→低**：依「體驗分數」欄位由高到低排序。
  - 邏輯在 `assets/js/studio.js` 的 `sortReviews()`，切換排序不會重新讀取試算表，只是把已經抓好的資料重新排序、重新渲染卡片。

- **噗浪圖示沒變粉紅色的排查**：這邊重新確認過 `assets/img/plurk.svg`，檔案內容是 `fill="#F0577E"`（粉紅），本身沒有問題。如果你部署後看到的還是黑色，通常是以下兩種情況：
  1. **repo 裡的檔案沒被換掉**：確認 GitHub 上 `assets/img/plurk.svg` 這個檔案的內容有沒有真的被這次的新版覆蓋過去（用 GitHub 網頁介面點開檔案直接看內容，開頭應該是 `<svg fill="#F0577E" ...>`）。
  2. **瀏覽器 / GitHub Pages 快取到舊檔**：SVG 圖片很容易被瀏覽器快取，就算 repo 裡的檔案已經換成新的，瀏覽器可能還在用舊版黑色圖示。可以試著強制重新整理（Windows/Linux：Ctrl+Shift+R；Mac：Cmd+Shift+R），或是換無痕視窗開網站確認。

## 本次調整

- **修正更新日誌讀不到的問題**：原本 `app.js` 在 `try` 區塊內用 `const` 宣告 `changelogRows`，但渲染的程式碼寫在 `try/catch` 外面，導致變數超出作用域、瀏覽器丟出錯誤，更新日誌永遠卡在「載入中」。現在改成在 `try` 區塊內直接呼叫 `renderChangelog()` 渲染，並把日期排序改成容錯的 `safeDateValue()`，避免試算表時間格式不同時整段當掉。

- **清掉沒在用的樣式**：例如 `.hero h1`／`.hero-eyebrow` 等首頁已經不會用到的區塊樣式。

## 上線前必做：開放試算表檢視權限

因為是純前端直接抓試算表資料，試算表**必須**設定為「知道連結的任何人 - 檢視者」，否則網站抓不到資料：

1. 開啟你的 Google 試算表
2. 右上角「共用」
3. 一般存取權設定為「知道連結的任何人」，角色為「檢視者」

> 這個做法只能「讀取」，訪客沒有辦法編輯或寫入你的試算表，資料仍然只有你能改。

## 部署到 GitHub Pages

1. 到 GitHub 建立一個新 repository（例如 `studio-guide`）
2. 把這整個資料夾的內容上傳（或用 git）：
   ```bash
   cd studio-site
   git init
   git add .
   git commit -m "init"
   git branch -M main
   git remote add origin https://github.com/你的帳號/studio-guide.git
   git push -u origin main
   ```
3. 到 repo 的 **Settings → Pages**
4. Source 選擇 `Deploy from a branch`，Branch 選 `main` / `root`
5. 存檔後等 1-2 分鐘，網站會出現在 `https://你的帳號.github.io/studio-guide/`

## 如果試算表欄位改了，改哪裡？

打開 `assets/js/config.js`，裡面的 `STUDIO_FIELDS` / `REVIEW_FIELDS` 對應的是試算表**表頭文字**，改欄位名稱或新增/刪除欄位時，同步調整這裡就好，不用動其他程式碼。

目前對應的分頁與欄位：

- **工作室列表**：工作室、噗浪帳號、官方網站、E-Mail、ICON、營運狀態、委託方式、工廠、客製項目、評價
  - `官方網站`、`E-Mail` 兩欄都可留白（留白就不會顯示對應圖示），表頭文字要跟 `config.js` 裡 `STUDIO_FIELDS.WEBSITE` / `STUDIO_FIELDS.EMAIL` 完全一致（注意 `E-Mail` 的大小寫）。
  - 已經沒有「平台」欄位，網站程式也已經拿掉對應的 badge。
- **體驗回報**：時間戳記、體驗者、工作室名稱、體驗分數、委印項目、體驗感想、返圖網址(1)、返圖網址(2)、返圖網址(3)、備註
  - 「電子郵件地址」欄位刻意**沒有**串接到網站上顯示，避免外洩體驗者信箱。
  - 三個返圖欄位都可留白，最多顯示 3 張縮圖；欄位名稱要跟 `config.js` 的 `REVIEW_FIELDS.IMAGE_URLS` 陣列裡的文字完全一致。
